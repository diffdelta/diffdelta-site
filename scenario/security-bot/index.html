The Security Bot Scenario

Your company needs to know when a vulnerability drops that affects your stack. Here\u2019s how that works today \u2014 and how it works with DiffDelta.

The Setup

You run a platform with a dozen services. Some are on Kubernetes, some on AWS. You depend on OpenSSL, Linux kernel, and a handful of open-source libraries. When a critical CVE drops, your team needs to know fast.

The sources you need to watch:

  • CISA Known Exploited Vulnerabilities \u2014 the US government\u2019s \u201cthese are being actively exploited right now\u201d list
  • NIST NVD \u2014 every published CVE with severity scores
  • GitHub Security Advisories \u2014 vulnerabilities in open-source packages you use
  • Kubernetes CVEs \u2014 if you run containers, you care about these
  • Linux Kernel CVEs \u2014 the operating system under everything
  • Ubuntu / Debian Security Notices \u2014 when your distro patches
  • OpenSSL Releases \u2014 the crypto library the internet depends on
  • AWS Service Health \u2014 is your infra even up?

That\u2019s 8 different sources, each with different formats, update frequencies, and access methods. Today, most teams handle this one of two ways \u2014 both painful.

The Old Way: Humans in the Loop

\u274c What it looks like today
  • Subscribe to 6 email lists (CISA, Ubuntu, Debian, Kubernetes, etc.)
  • Check the NVD website manually every morning
  • Skim GitHub\u2019s advisory feed when you remember
  • Get 300+ emails/day, 95% of which don\u2019t affect your stack
  • Your security engineer spends 1\u20132 hours/day triaging \u2014 reading, correlating, deciding what matters
  • Critical CVEs occasionally slip through because someone was on PTO
\u274c What it looks like with DIY bots
  • Build 4\u20136 custom scrapers (one per source)
  • Parse HTML, RSS, JSON, and CSV \u2014 every source is different
  • Handle rate limits, pagination, and format changes
  • Your LLM processes 43MB of raw data to find what changed
  • Scrapers break every few weeks when upstream sites update
  • Maintaining this is a 200\u2013400 line codebase per source

Both approaches work, sort of. But they\u2019re expensive, fragile, and don\u2019t scale. Every new source is another scraper to build, another email to parse, another thing that breaks.

The DiffDelta Way: Agent-Ready Intelligence

With DiffDelta, all 8 of those sources are already being monitored. Every 15 minutes, our pipeline checks upstream, computes what changed, scores it for risk, and serves it as structured JSON on a global CDN.

Your bot\u2019s job becomes trivial:

\u2705 What it looks like with DiffDelta
  • Poll one endpoint every 15 minutes: /diff/head.json (400 bytes)
  • If nothing changed, stop. You burned 400 bytes, not 43MB.
  • If something changed, fetch /diff/latest.json \u2014 pre-diffed, pre-normalized JSON
  • Every item has a headline, excerpt, risk score, and provenance chain
  • Filter by source (cisa_kev, github_advisories, etc.) or tag (security)
  • Your agent reads 49K tokens instead of 43M \u2014 and knows exactly what\u2019s new

security_bot.py
import requests, json

# Your stored cursor (start with None on first run)
stored_cursor = load_cursor()  # e.g. from a file or database

# Step 1: Check for changes (400 bytes, ~10ms)
head = requests.get("https://diffdelta.io/diff/head.json").json()

if head["cursor"] == stored_cursor:
    print("No changes. Done.")
    exit()

# Step 2: Fetch the full delta
feed = requests.get("https://diffdelta.io/diff/latest.json").json()
